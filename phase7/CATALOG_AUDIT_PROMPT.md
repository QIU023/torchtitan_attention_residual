# Catalog audit prompt — `phase7/pattern_catalog.md` vs phase5 run logs

This file is meant to be fed verbatim as the prompt for an OMC worker
(or any other fresh `claude` CLI session) that should not have to read
this conversation's history. The worker only needs file-system access
to `/root/torchtitan_attention_residual/`.

## Usage

```bash
# In a session where /omc-teams is configured:
/omc-teams 2:claude "$(cat /root/torchtitan_attention_residual/phase7/CATALOG_AUDIT_PROMPT.md)"

# Or paste the prompt body below into any other tool.
```

## Prompt body (copy from here on)

Audit `phase7/pattern_catalog.md` against `phase5/runs/8gpu_*/train.log`
and flag inconsistencies. **Do not run any GPU work; do not modify
`pattern_catalog.md` directly.** Output one new file:
`phase7/CATALOG_AUDIT.md`.

### Background (must read first to avoid duplicating prior diagnosis)

`phase7/pattern_catalog.md` was auto-generated by
`phase7/build_pattern_catalog.py` from `collective_summary.csv` files
that live in `phase5/runs/8gpu_*/tier_{a,b,c}_trace/` directories. The
generator script ingests every CSV it finds. The CSVs themselves are
produced by `phase7/extract_collectives.py` parsing the raw NCCL
`nccl-rank-*.log` files written by torchrun.

The catch: **NCCL writes its INFO log during model init too**, before
any training step runs. Many runs in `phase5/runs/8gpu_*/` failed during
init (TP plan errors, mesh-arithmetic mistakes, dtype mismatches).
Their tier_c_trace dirs still contain log files (init-phase collectives
only) and their CSVs still exist, so the catalog inherits init-only
"signatures" as if they were real workload traces.

The previous human-driven session's reality is captured in
`phase6/SESSION_8GPU_summary.md` — read that first. Bottom line: only
two configs reached step ≥ 1 of training:

- **B0 (FSDP=8 PP=1)** — full 500-step alignment, GBS=16 LBS=2
- **A2 (FSDP=2 × PP=4 V=2 + cache adapter)** — full 500-step alignment, GBS=16 LBS=1

Plus B0 has Tier B (GBS=120, 50 steps) and Tier A (GBS=384, 100 steps)
traces that did complete. Everything else is either init-only or empty.

### Audit rules

For every (config, tier) row in `phase7/pattern_catalog.md`:

1. Locate the corresponding train.log:
   - For tier_c: `phase5/runs/8gpu_<cfg>_seed42/train.log`
   - For tier_a / tier_b: `phase5/runs/8gpu_<cfg>_tier_{a,b}/train.log`

2. Strip ANSI color codes:
   `sed 's/\x1b\[[0-9;]*m//g' train.log`

3. A "real" run = at least one line matches:
   `step:[ ]+[0-9]+[ ]+loss:[ ]+[0-9.-]+`

   Anything else (only `Traceback`, only NCCL bootstrap warnings, only
   "Training starts at step 1" with no subsequent step) = **init-only,
   not a real workload trace.**

4. Cross-check against the NCCL trace dir:
   `phase5/runs/.../tier_{a,b,c}_trace/nccl-rank-*.log` — count lines
   matching `opCount [0-9]+ .* count [0-9]+`. If `count` per call is
   universally < 1024 elements (i.e. only kBytes-scale collectives),
   that's also init-only — real forward/backward fires multi-MB
   collectives in dense MLP / FSDP all-gather paths.

### Output: `phase7/CATALOG_AUDIT.md`

A markdown report with these sections in order:

1. **`## Summary`** — one-line per (config, tier): `OK` / `INIT-ONLY` / `MISSING-LOG` / `EMPTY-LOG`.

2. **`## Real runs (keep in catalog)`** — each entry shows:
   - run dir
   - last training step reached (and final loss if available)
   - one-line collective-summary recap (top 3 op×size_bucket from CSV)

3. **`## Init-only or missing runs (recommend filter from catalog)`** —
   each entry shows:
   - run dir
   - reason (init-only / empty log / missing log)
   - what the catalog currently says about it (so the human can confirm
     the entry should be removed or annotated `[FAILED-INIT]`)

4. **`## Recommendation for build_pattern_catalog.py`** — a concrete
   patch suggestion: add a function `_run_reached_training(run_dir)`
   that reads `train.log` and returns `True` only if the regex above
   matched at least once. Skip CSVs whose corresponding run failed
   the check. Show the diff in unified-diff format pointing at
   `phase7/build_pattern_catalog.py`.

5. **`## Suggested annotation for surviving entries`** — for the kept
   entries, suggest a header line in `pattern_catalog.md` explaining
   that all rows passed the "reached training" filter and listing the
   exact filter regex used.

### Constraints

- **Read-only on `phase5/runs/`**. Do not delete or move any files.
- **Do not modify `pattern_catalog.md` directly** — your output is the
  audit report, not a rewrite.
- **Do not run any GPU work** (no torchrun, no training).
- It is OK to run `phase7/extract_collectives.py` against existing
  trace dirs to re-derive histograms if a CSV is stale, but the
  underlying NCCL log files themselves must not be modified.
- Use `git status` and `git log --oneline -5` early to confirm you're
  reading from the latest committed state. If the workspace has
  uncommitted local edits, note them at the top of your audit report
  but do not act on them.

### Done definition

`phase7/CATALOG_AUDIT.md` exists, contains all 5 sections, and the
"Real runs" + "Init-only or missing runs" lists together cover every
(config, tier) row currently in `pattern_catalog.md`. No GPU was
touched. No file under `phase5/runs/` was modified.
